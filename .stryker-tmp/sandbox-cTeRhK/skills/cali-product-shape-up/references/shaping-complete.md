# Shape Up — Context, Clarification & Responsibilities

## Context Reconstruction

Before asking questions:

1. **Infer as much context as possible** from:
   - current request
   - session history
   - prior brainstorming
   - referenced plans/specs
   - implied workflows
   - business/operational signals

2. **Reconstruct internally:**
   - likely goals
   - stakeholders
   - tensions
   - workflows
   - assumptions
   - constraints
   - likely job-to-be-done

Avoid asking users to restate existing context.

---

## Clarification Rules

Prefer:
- assumptions
- iterative shaping
- provisional framing

over blocking for perfect information.

### **Prefer**: assumptions, iterative shaping, provisional framing

Ask questions only when missing information materially changes:
- scope
- feasibility
- sequencing
- operational viability
- workflow structure
- risk analysis

### Avoid:
- asking users to restate known context
- exhaustive requirement gathering
- unnecessary confirmation loops

---

## Question Tool Usage

When available:
- always use `question`

Questions should:
- include explicit options
- include one AI-recommended option
- explain recommendation briefly
- remain overridable

**Example:**
- A — broader MVP
- B — narrow workflow-first scope
- R — recommended: narrow workflow-first scope to reduce integration risk

If unavailable:
- use equivalent chat structure.

Questions should: include explicit options, include one AI-recommended option, explain recommendation briefly, remain overridable.

## Main Responsibilities

The shaping phase should:
1. Clarify the problem
2. Reveal assumptions
3. Expose risks
4. Define scope boundaries
5. Identify linchpins
6. Shape coherent solution direction
7. Prevent scope explosion
8. Prepare downstream UX and implementation work
