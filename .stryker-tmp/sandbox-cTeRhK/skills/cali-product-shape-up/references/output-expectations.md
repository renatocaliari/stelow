# Output Expectations

## Strong vs Weak Output Criteria

### Strong Output
- Problem clearly defined: user, context, job-to-be-done
- IN/OUT scope explicit and justified
- Risks mapped with probability and impact
- Strategic alternatives considered and dismissed with rationale
- Interface proposal (if applicable) with wireframes, breadboarding, trade-offs
- Plan Critique with gaps classified by severity
- Formal approval registered (approved: true + receipt)
- Typed scopes with DoD, acceptance criteria, dependencies
- Execution routing: each scope mapped to executor

### Weak Output
- Vague or unclarified problem
- Poorly defined scope or no justification
- Risks not mentioned
- Alternatives ignored
- Interface without wireframes or breadboarding
- Critique without gaps or classification
- Verbal approval without formal registration
- Scopes without type, DoD, or acceptance criteria
- No execution routing
