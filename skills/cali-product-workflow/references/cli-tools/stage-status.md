# Stage Status Helper

> Utility to update workflow stage progress for human reviewers.

---

## Commands

| Command | Description |
|---------|-------------|
|| `/pw-setphase stage=N` | Set current stage by index (0=Setup, 1=Context, ...) or `phasename=Name` |
|| `/pw-next` | Advance to next stage |
|| `/pw-status` | Show workflow status |

---

## Stage Index (0-indexed for CLI)

| Index | Stage Name | Workflow Stage |
| 0 | Setup | Stage 1 |
| 1 | Context | Stage 2 |
| 2 | Shape | Stage 3 |
| 3 | Interface | Stage 4 |
| 4 | Critique | Stage 5 |
| 5 | Gate | Stage 6 |
| 6 | Scope Adjustment | Stage 7 |
| 7 | Interface Gate | Stage 8 |
| 8 | Interface Selection | Stage 9 |
| 9 | Tech Planning | Stage 10 |
| 10 | Execution | Stage 11 |
| 11 | Verification | Stage 12 |
| 12 | Execution Critique | Stage 13 |
| 13 | Complete | Workflow Done |

---

## Examples

```bash
# Show current status
/pw-status

# Set to Shape stage
/pw-setphase phasename=Shape

# Advance to next stage
/pw-next
```

## Auto-Update vs Manual

**Auto-update:** The extension updates `currentStage` when user uses `/pw-next`, `/pw-setphase`.

**Manual update:** If the LLM completes a stage but the extension didn't update (e.g., LLM ran the skill standalone), the user can manually update:

```bash
# Prefer phasename (immune to reordering):
/pw-setphase phasename=Shape

# Or use 0-indexed stage number:
/pw-setphase stage=2
```

## Verification (after Execution)

After all scopes are executed, the Verification stage runs:
1. **Run test suite** — block until tests pass
2. **Parallel code review** — if 3+ files touched
3. **UI quality** — if visual interface
4. **Browser testing** — if interactive
5. **Final checklist** — pre-completion verification

See `stages/verification.md` for the full protocol.

## Execution Critique (after Verification)

After Verification passes, the Execution Critique stage verifies:

1. **Scope completeness** — all planned scopes implemented
2. **Gap analysis** — identify missing or extra features
3. **Quality gates** — tests passing, lint clean, build succeeds
4. **Lessons learned** — document what went well/what could improve

Delegates to `skills/cali-product-execution-critique/SKILL.md` — outputs `execution-critique.md` with findings and decision matrix.
